# Contribution 09 — Multi-Robot Coordination under Uncertainty

[![Module](https://img.shields.io/badge/Module-09-purple)](.) [![Type](https://img.shields.io/badge/Type-Multi--Robot%20Systems-blue)](.) [![Status](https://img.shields.io/badge/Status-Core%20Upgraded-brightgreen)](.)

## Plain-language summary

A group of robots should not behave like independent robots that accidentally share the same environment.

They need to coordinate:

```text
Where is each robot going?
Will two robots collide or swap positions at the same time?
Is each robot staying within its risk budget?
Do all robots agree about the map?
```

Contribution 09 studies decentralized multi-robot coordination under uncertainty. The upgraded version adds explicit metrics for team feasibility, path conflicts, risk-budget violations, and belief disagreement.

---

## Research Question

> **RQ9:** How can multiple robots coordinate navigation under uncertainty without relying on a single central coordinator?

This contribution studies:

- decentralized path coordination,
- map and belief sharing,
- spatial-temporal path conflicts,
- team risk allocation,
- belief disagreement,
- team-level feasibility.

---

## Motivation

Multi-robot navigation can fail even when every individual robot has a valid path.

Examples:

- two robots plan to occupy the same cell at the same time,
- two robots swap positions across the same edge at the same time,
- one robot consumes more than its allocated risk budget,
- one robot has a map that disagrees with the rest of the team,
- communication delay creates inconsistent planning assumptions.

C09 makes these failures measurable.

---

## Conceptual Pipeline

```text
N robot local plans
      ↓
shared belief / map summary
      ↓
path conflict detection
      ↓
risk-budget check
      ↓
belief-disagreement check
      ↓
team feasibility verdict
```

---

## Coordination Strategies

| Strategy | Description | Scalability |
|---|---|---|
| Centralized | Single coordinator plans all paths | Low |
| Decentralized priority | Robots reserve paths by priority | Medium |
| Market-based | Robots bid for frontiers or path segments | High |
| Consensus-based | Robots agree on shared plan or belief | Medium–High |

---

## Metrics Added

| Metric | Meaning |
|---|---|
| Vertex conflict | Two robots occupy the same location at the same time |
| Edge-swap conflict | Two robots swap positions across the same edge at the same time |
| Risk-budget violation | A robot's planned risk exceeds its allocated budget |
| Belief disagreement | One robot's belief summary differs from the team majority |
| Team feasibility | True only when no conflicts, no budget violations, and no belief disagreement are detected |

## New Benchmark

The benchmark evaluates four small but interpretable scenarios:

| Scenario | Purpose |
|---|---|
| `conflicting_paths` | Detects vertex and edge-swap conflicts |
| `risk_budget_violation` | Detects robot-level risk overuse |
| `belief_disagreement` | Detects inconsistency in shared belief summaries |
| `coordinated_feasible` | Shows a clean team plan with no detected violations |

## Scientific Contribution

The upgraded C09 contribution is not simply:

> Multiple robots share maps and coordinate.

It is stronger:

> Multi-robot coordination is evaluated through explicit team-level feasibility metrics covering path conflicts, risk-budget allocation, and belief consistency.

This makes the module auditable and easier to connect to later swarm consensus and fault-tolerance modules.

## Limitations

- The current benchmark uses discrete time and grid paths.
- Belief disagreement is represented by compact hashes, not full probabilistic map comparison.
- Risk budgets are scalar and do not yet capture spatial or temporal risk distribution.
- Real deployment should model communication delay, packet loss, ROS 2 namespaces, kinodynamics, and decentralized replanning.

---

## Next Research Step

The strongest next extension is decentralized repair:

```text
conflict / risk violation / belief disagreement
      ↓
local negotiation
      ↓
revised team plan
```

This would turn C09 from a coordination evaluator into a decentralized multi-robot replanning module.
