# Contribution 08 — Security, Intrusion Detection, and Trust-Aware Response

[![Module](https://img.shields.io/badge/Module-08-purple)](.) [![Type](https://img.shields.io/badge/Type-Cybersecurity%20%2F%20IDS-blue)](.) [![Status](https://img.shields.io/badge/Status-Core%20Upgraded-brightgreen)](.)

## Plain-language summary

A robot should not blindly trust its sensors, localization, or external signals.

Contribution 08 adds an anomaly-based intrusion detection system for robot navigation. It monitors innovation residuals and detects abnormal behaviour that may indicate spoofing, drift, sensor failure, or adversarial interference.

The upgraded version goes one step further:

> It converts IDS evidence into trust scores, alert severities, and planner mitigation actions.

This means the IDS is not only a diagnostic module. It becomes a navigation safety input.

---

## Research Question

> **RQ8:** How can an autonomous navigation system detect degraded trust and convert cyber-physical anomalies into safe planner responses?

This contribution studies:

- innovation monitoring,
- Mahalanobis anomaly scores,
- chi-square thresholding,
- consecutive and k-of-n trigger policies,
- trust scoring,
- alert severity classification,
- safe-mode and emergency-stop recommendations.

---

## Motivation

Navigation systems can fail when the robot receives unreliable information.

Examples:

- GPS spoofing shifts the estimated pose,
- IMU drift accumulates slowly,
- LiDAR or depth data becomes corrupted,
- map updates are manipulated,
- communication links inject inconsistent information,
- localization residuals become statistically abnormal.

A planner should know when the information it uses is no longer trustworthy.

## Conceptual Pipeline

```text
sensor / estimator output
      ↓
innovation residual
      ↓
Mahalanobis distance / chi-square test / CUSUM-like trigger
      ↓
IDS alert
      ↓
trust score + alert severity
      ↓
planner mitigation: caution, safe mode, or emergency stop
```

## Detection Methods

| Method | Best For | Typical latency |
|---|---|---|
| χ² / Mahalanobis test | Single-step anomalies | 1 step |
| Consecutive trigger | Sustained abnormal residuals | few steps |
| k-of-n trigger | Noisy repeated anomalies | configurable |
| CUSUM-style monitoring | Gradual spoofing / drift | 5–20 steps |

## Existing Core Monitor

The existing monitor computes an innovation anomaly score:

```text
d² = νᵀ S⁻¹ ν
```

where:

- `ν` is the innovation residual,
- `S` is the innovation covariance,
- `d²` is the Mahalanobis squared distance.

The score is compared with an approximate chi-square threshold.

If anomaly evidence persists, the monitor triggers an IDS alert.

## New Upgrade Added

C08 now includes:

```text
code/ids_response_policy.py
```

This maps IDS monitor outputs to:

- alert severity,
- trust score,
- planner mitigation,
- explanation.

### Alert levels

| Severity | Meaning | Planner response |
|---|---|---|
| `NORMAL` | Innovation is expected | No mitigation |
| `WATCH` | Innovation is elevated | Increase caution |
| `WARNING` | IDS trigger or high anomaly evidence | Trigger safe mode |
| `CRITICAL` | Sustained or extreme anomaly | Emergency stop |

## Limitations

- The response policy is currently rule-based.
- Thresholds should be validated per sensor and estimator.
- Synthetic benchmark outputs are useful for auditability but not enough for final deployment claims.
- Emergency stop is represented as a planner-level mitigation recommendation, not a certified hardware stop.
- IDS detection does not by itself identify root cause; C14 addresses attribution.

---

## Next Research Step

The strongest next extension is attack-conditioned mitigation:

```text
mitigation = f(sensor_type, anomaly_source, d2_ratio, attack_duration, navigation_context)
```

This would allow the system to respond differently to GPS spoofing, IMU drift, LiDAR corruption, map tampering, or communication attacks.
